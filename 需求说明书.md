# 基于多智能体协同的果蔬病害检测与智能防治系统 -需求说明书



# 1 项目背景与目标

## 1.1 项目背景

&emsp;&emsp;随着现代农业的发展，果蔬种植规模不断扩大，病害已成为影响果蔬产量、品质及经济效益的重要因素。苹果黑星病、番茄晚疫病、葡萄黑腐病、玉米锈病等病害具有传播速度快、危害范围广、识别难度高等特点，一旦发现不及时，将导致大面积减产甚至绝收。

&emsp;&emsp;目前，大多数农业生产仍主要依赖人工巡查进行病害识别，存在以下不足：

- **检测效率低**：人工巡查需要逐株观察叶片，耗时耗力，难以满足大规模种植基地的日常巡检需求。
- **识别准确率受经验影响**：不同病害症状相似，基层种植人员难以准确区分各种病害类型，容易发生误诊和漏诊。
- **缺乏实时预警能力**：传统方式无法实现病害发生情况的实时监测，容易错过最佳防治时期。
- **数据管理分散**：病害检测、历史记录、模型训练及统计分析相互独立，缺乏统一的信息管理平台。
- **智能化程度不足**：传统农业信息系统难以结合人工智能技术，为用户提供病害分析、防治建议及数据决策支持。

&emsp;&emsp;近年来，随着深度学习目标检测技术的发展，以 YOLO 系列模型为代表的实时目标检测算法在农业病害识别领域取得了良好的应用效果。结合高质量植物病害数据集，可实现果蔬叶片病害的快速定位、精准识别和智能分析，为智慧农业提供可靠的技术支撑。

&emsp;&emsp;因此，本项目拟设计并实现一套基于YOLOv11的果蔬病害智能检测系统，实现果蔬叶片病害检测、模型训练、数据管理、可视化分析以及 AI 智能问答等功能，提升农业病害检测的自动化和智能化水平。



## 1.2 项目目标

&emsp;&emsp;本项目以 Plant Disease Detection 数据集为基础，构建一套集数据管理、模型训练、目标检测、数据分析和智能问答于一体的果蔬病害智能检测平台。

&emsp;&emsp;项目总体目标如下：

| 层面 | 建设目标 |
|------|----------|
| 数据层 | 支持果蔬病害数据集导入、管理、预览、统计及数据增强，满足YOLO训练要求。 |
| 模型层 | 基于YOLOv11完成30类果蔬叶片病害目标检测模型训练、评估及部署。 |
| 检测层 | 支持图片、视频、摄像头等多种检测方式，实现病害实时识别。 |
| 平台层 | 构建前后端分离的智能检测平台，实现用户管理、权限管理、检测管理、AI问答及数据可视化。 |
| 应用层 | 为农业种植户、农技人员及科研人员提供病害识别、防治建议和数据统计分析服务。 |



## 1.3 系统建设目标

（1）建立标准化病害数据管理平台

&emsp;&emsp;实现果蔬病害数据集的统一导入、存储、管理、预览及统计分析，提高数据管理效率。

（2）建立高精度病害检测模型

&emsp;&emsp;利用YOLOv11完成植物病害检测模型训练，提高模型在复杂农业环境下的检测精度和实时性。

（3）实现智能病害检测

&emsp;&emsp;支持单图检测、批量检测、视频检测及摄像头实时检测，实现病害目标自动定位与分类。

（4）构建智能辅助决策平台

&emsp;&emsp;结合大语言模型（LLM）技术，根据检测结果自动生成病害介绍、防治方法、发生原因及农业管理建议。

（5）实现检测数据可视化

&emsp;&emsp;统计病害发生次数、病害分布情况、模型精度、检测记录等信息，为农业生产提供数据支撑。

# 2 系统总体需求

## 2.1 系统总体架构

&emsp;&emsp;本系统采用前后端分离的 B/S 架构，面向农业生产中的果蔬病害检测场景。前端为 Vue 3 单页应用（SPA），通过 Axios 调用后端 RESTful API。后端 FastAPI 提供 API 服务和 WebSocket 实时通信，通过 SQLAlchemy ORM 访问 PostgreSQL 数据库，通过 MinIO SDK 管理对象存储。所有基础设施（PostgreSQL、Redis、MinIO）通过 Docker Compose 容器化部署。

系统主要由以下模块组成：

- 用户管理模块
- 权限管理模块
- 数据集管理模块
- 模型训练模块
- 果蔬病害检测模块
- AI智能问答模块
- 数据可视化模块
- 历史记录管理模块

### 3.1.3 角色权限设计说明

| 设计要点 | 说明 |
|---------|------|
| **最小权限原则** | 普通用户仅拥有完成本职工作所需的最小权限集 |
| **数据隔离** | 普通用户只能访问自己创建的数据（数据集、训练任务、检测记录等） |
| **管理员豁免** | 管理员拥有全部功能权限和跨用户数据访问权限 |
| **可扩展性** | 采用RBAC模型设计，未来可灵活新增角色（如审计员、访客等） |
| **系统内置角色** | 管理员和普通用户为系统内置角色，不可删除 |
| **权限细分** | 采用细粒度权限控制（如 detection:task:create），支持灵活组合 |

## 3.4 果蔬病害检测模块
### 3.4.1 模块说明

支持多种检测方式：

（1）图片检测：上传单张图片完成病害检测。

（2）批量检测：一次上传多张图片进行批量检测。

（3）文件夹检测：支持整个数据集批量检测。

（4）视频检测：上传视频进行逐帧病害检测。

（5）摄像头检测：调用本地摄像头进行实时检测。

检测结果包括：

- 病害名称
- 病害位置
- 置信度
- 检测框
- 检测数量

支持结果保存及导出。

### 3.4.2 检测结果说明

| 返回字段 | 说明 |
|---------|------|
| 病害名称 | 病害的中英文名称 |
| 病害位置 | 检测框坐标（x1,y1,x2,y2） |
| 置信度 | 模型预测置信度（0-1） |
| 检测框 | 标注框在原图中的位置 |
| 检测数量 | 本次检测到的目标总数 |
| 检测结果图 | 带标注框的结果图片URL |
| 防治建议 | AI生成的防治建议（可选） |

## 5.4 数据集特点


（1）类别丰富

&emsp;&emsp;共包含30个检测类别，覆盖9种以上主要经济作物及19种典型植物病害，可满足多类别目标检测任务。

（2）真实农业场景

&emsp;&emsp;图片采集于真实种植环境，背景复杂，包括：土壤背景、天空背景、枝干背景、杂草背景、多叶片重叠、光照变化、阴影遮挡等。相比实验室采集数据，更符合实际农业应用需求。

（3）目标尺度差异大

&emsp;&emsp;病斑面积存在较大差异，包括：小面积病斑、中等病害区域、大面积感染叶片、整片叶片感染。


（4）类别相似度较高

&emsp;&emsp;部分病害具有较高相似性，例如：番茄早疫病、番茄晚疫病、番茄叶霉病。因此容易出现误检，可以验证具有较强特征提取能力。

（5）健康叶片与病害叶片共存

&emsp;&emsp;系统不仅需要识别病害，还需要正确判断健康叶片，降低误报率，提高实际应用价值。


# 6 API接口需求

## 6.1 接口设计原则

为了保证系统模块之间的高效协作与后期扩展性，本系统采用 RESTful API 设计规范，前后端通过 HTTP/HTTPS 协议进行通信。

接口设计遵循以下原则：

- 统一使用 JSON 数据格式进行数据交换
- 统一返回标准响应结构
- 统一使用 JWT 进行身份认证
- 统一异常处理机制
- 支持分页查询
- 支持文件上传与下载
- 支持 SSE 实时推送
- 支持 WebSocket 实时通信



### 6.1.1 接口基础路径

```text
http://localhost:8000/api/v1
```

生产环境：

```text
https://domain/api/v1
```

### 6.1.2 统一响应格式

#### 成功响应

```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

#### 失败响应

```json
{
  "code": 500,
  "msg": "error",
  "data": null
}
```

```http

```

除登录与注册接口外，其余接口均需携带 Token。

## 7. 非功能性需求

### 7.1 性能需求
| 指标 | 要求 |
|------|------|
| 图片检测响应时间 | ≤200ms |
| 视频检测帧率 | ≥20FPS |
| API接口响应时间 | ≤500ms（不含AI推理） |
| 系统并发用户数 | ≥50人 |
| 数据库查询响应 | ≤100ms |

### 7.2 安全需求
- 用户密码采用bcrypt加密存储
- JWT Token有效期30分钟，支持刷新
- 操作日志保留至少90天
- 文件上传需校验文件类型和大小

### 7.3 可用性需求
- 关键操作（检测、训练）支持失败重试
- 系统异常时返回友好的错误提示

### 7.4 可扩展性需求
- 支持新增病害类别（30类基础上扩展）
- 支持接入新的YOLO模型版本
- 支持多数据源接入

## 8. 系统部署需求

### 8.1 PC端部署

支持操作系统：Windows、Linux。
推荐方式：Python + FastAPI + Docker。

### 8.2 Web端部署

前端：Vue3 + Vite，通过 npm run build 打包为静态文件。
后端：通过 uvicorn 启动 API 服务。
一体化部署：使用 docker-compose 编排前后端、PostgreSQL、Redis、MinIO 等服务。

### 8.3 Docker Compose 服务编排

PostgreSQL（端口5432）：业务数据主库 + Pgvector向量扩展。
Redis（端口6379）：缓存与对话历史。
MinIO（端口9000/9001）：图片与模型文件存储。
Backend（端口8000）：FastAPI后端服务。
Frontend（端口80）：Vue3前端静态服务。
Ollama（端口11434）：本地LLM备选（可选）。
