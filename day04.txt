⼆、Python logging 模块配置与⽇志分级
2.1 为什么需要⽇志系统
在 Day 2 中，我们⽤ print() 输出信息。但在企业级项⽬中， print() 存在严重不⾜：


 ┌────────────────────────────────────────────────────────────────┐
 │                    print() vs logging 对⽐                           │
 ├──────────────────────┬─────────────────────────────────────────┤
 │       特性               │           对⽐结果                        │
 ├──────────────────────┼─────────────────────────────────────────┤
 │ ⽇志分级                  │ print ⽆法区分 INFO/WARNING/ERROR           │
 │                         │ logging ⽀持 DEBUG/INFO/WARNING/ERROR       │
 ├──────────────────────┼─────────────────────────────────────────┤
 │ 输出⽬标                  │ print 只能输出到终端                           │
 │                         │ logging 可同时输出到终端+⽂件+远程服务              │
 ├──────────────────────┼─────────────────────────────────────────┤
 │ ⽇志格式                  │ print 需要⼿动拼接时间、模块名                  │
 │                         │ logging ⾃动附加时间戳、模块、⾏号                 │
 ├──────────────────────┼─────────────────────────────────────────┤
 │ ⽂件轮转                  │ print 不⽀持                               │
 │                         │ logging ⽀持按时间/⼤⼩⾃动切割⽇志⽂件              │
 ├──────────────────────┼─────────────────────────────────────────┤
 │ ⽣产可控                  │ print ⽆法在运⾏时调整输出级别                  │
 │                         │ logging 可通过配置⽂件动态控制                       │
 └──────────────────────┴─────────────────────────────────────────┘



2.2 ⽇志级别说明
Python logging 提供 5 个标准级别，按严重程度递增：


 级别           数值     使⽤场景
 ─────────────────────────────────────────────────────────
 DEBUG         10     开发调试信息（变量值、函数调⽤）
 INFO          20     正常运⾏信息（服务启动、请求处理）
 WARNING       30     潜在问题（配置缺失⽤默认值、连接重试）
 ERROR         40     错误事件（请求失败、数据库异常）
 CRITICAL      50     严重错误（服务崩溃、磁盘空间不⾜）


原则：开发环境⽤ DEBUG ，⽣产环境⽤ INFO ，只输出当前级别及以上的⽇志。

2.3 创建⽇志配置模块
⽂件： backend/app/core/logger.py


 """
 统⼀⽇志配置模块


 职责：
     - 配置全局⽇志格式和输出级别
     - 同时输出到控制台和⽇志⽂件
     - ⽇志⽂件按⼤⼩⾃动轮转（保留最近 N 份）


 使⽤⽅式：
     在其他模块中引⼊：
        from app.core.logger import get_logger
        logger = get_logger(__name__)
        logger.info("服务启动成功")
 """


 import logging
import os
import sys
from logging.handlers import RotatingFileHandler


from app.config.settings import settings


# ── ⽇志⽬录 ──────────────────────────────────────────
# ⽇志⽂件存放在 backend/logs/ ⽬录下
LOG_DIR = os.path.join(os.path.dirname(os.path.dirname(os.path.dirname(__file__))),
"logs")
os.makedirs(LOG_DIR, exist_ok=True)


# ── ⽇志格式 ──────────────────────────────────────────
# 格式说明：
#   %(asctime)s        — 时间戳（精确到毫秒）
#   %(levelname)-8s — ⽇志级别（左对⻬，8 字符宽）
#   %(name)s           — Logger 名称（通常是模块名）
#   %(funcName)s       — 调⽤⽇志的函数名
#   %(lineno)d         — 调⽤⽇志的代码⾏号
#   %(message)s        — ⽇志内容
LOG_FORMAT = "%(asctime)s | %(levelname)-8s | %(name)s:%(funcName)s:%(lineno)d | %
(message)s"
DATE_FORMAT = "%Y-%m-%d %H:%M:%S"


# ── 格式化器 ──────────────────────────────────────────
formatter = logging.Formatter(LOG_FORMAT, datefmt=DATE_FORMAT)


# ── 是否已初始化标记（避免重复添加 handler）────────────
_initialized = False



def setup_logging():
    """
    初始化全局⽇志系统（只需调⽤⼀次）


    配置两个输出⽬标：
      1. 控制台（StreamHandler）— 开发时实时查看
      2. ⽂件（RotatingFileHandler）— 持久化存储，按⼤⼩轮转
    """
    global _initialized
    if _initialized:
          return
    _initialized = True


    # 获取根 Logger
    root_logger = logging.getLogger()
    root_logger.setLevel(getattr(logging, settings.LOG_LEVEL, logging.INFO))


    # ── 控制台 Handler ────────────────────────────────
    console_handler = logging.StreamHandler(sys.stdout)
    console_handler.setLevel(logging.DEBUG)
    console_handler.setFormatter(formatter)
     root_logger.addHandler(console_handler)


     # ── ⽂件 Handler（轮转）──────────────────────────
     # maxBytes=10MB：单个⽇志⽂件超过 10MB 时⾃动切割
     # backupCount=5：保留最近 5 份历史⽇志
     # encoding="utf-8"：确保中⽂⽇志正常显示
     file_path = os.path.join(LOG_DIR, "app.log")
     file_handler = RotatingFileHandler(
           filename=file_path,
           maxBytes=10 * 1024 * 1024,   # 10MB
           backupCount=5,
           encoding="utf-8",
     )
     file_handler.setLevel(logging.INFO)     # ⽂件只记录 INFO 及以上
     file_handler.setFormatter(formatter)
     root_logger.addHandler(file_handler)


     # ── 降低第三⽅库⽇志级别，避免刷屏 ──────────────────
     logging.getLogger("uvicorn").setLevel(logging.WARNING)
     logging.getLogger("sqlalchemy").setLevel(logging.WARNING)
     logging.getLogger("minio").setLevel(logging.WARNING)
     logging.getLogger("httpx").setLevel(logging.WARNING)



 def get_logger(name: str) -> logging.Logger:
     """
     获取指定名称的 Logger


     Args:
           name: Logger 名称，通常传⼊ __name__（模块路径）


     Returns:
           配置好的 Logger 实例


     使⽤示例：
           logger = get_logger(__name__)
           logger.info("⽤户登录成功: user_id=%d", user_id)
           logger.error("数据库连接失败: %s", error_msg)
     """
     setup_logging()    # 确保⽇志系统已初始化
     return logging.getLogger(name)


关键知识点讲解：

 1. RotatingFileHandler ：⾃动轮转⽇志⽂件。当⽇志⽂件超过 maxBytes （10MB）时，⾃动重命名为
   app.log.1 ，新⽇志写⼊ app.log 。最多保留 backupCount （5）份，⾃动删除最旧的

 2. get_logger(__name__) ：传⼊ __name__ 是 Python 最佳实践。Logger 名称⾃动反映模块路径，如
   app.api.auth ，⽅便定位⽇志来源

 3. 降低第三⽅库⽇志级别：uvicorn、sqlalchemy 等库的 DEBUG ⽇志⾮常多，设为 WARNING 避免⼲扰

三、后端全局异常处理中间件
3.1 为什么需要全局异常处理
没有全局异常处理时，未捕获的异常会返回 FastAPI 默认的 500 Internal Server Error ，前端只看到⼀个空泛
的错误，⽆法定位问题。


 ┌─────────────────────────────────────────────────────────────┐
 │                        全局异常处理流程                                     │
 │                                                                   │
 │     API 请求 ──► 业务代码执⾏                                         │
 │                        │                                          │
 │                        ├── 正常 ──► 返回 JSON 响应                    │
 │                        │                                          │
 │                        └── 抛出异常                                   │
 │                            │                                      │
 │                            ▼                                      │
 │                        全局异常处理器                                    │
 │                            │                                      │
 │                            ├── HTTPException ──► 返回原始状态码          │
 │                            ├── ValidationError ──► 422 参数错误       │
 │                            └── Exception ──► 500 + 记录⽇志           │
 │                                                                   │
 │     所有异常都返回统⼀的 JSON 格式：                                       │
 │     {                                                             │
 │         "code": 500,                                              │
 │         "message": "服务器内部错误",                                     │
 │         "detail": "具体错误描述（仅 DEBUG 模式）"                          │
 │     }                                                             │
 └─────────────────────────────────────────────────────────────┘



3.2 创建全局异常处理模块
⽂件： backend/app/core/exceptions.py


 """
 全局异常处理模块


 职责：
     - 捕获所有未处理的异常，返回统⼀ JSON 格式
     - 区分已知异常（HTTP 业务错误）和未知异常（程序 Bug）
     - 未知异常⾃动记录 ERROR 级别⽇志


 注册⽅式（在 main.py 中）：
     from app.core.exceptions import register_exception_handlers
     register_exception_handlers(app)
 """


 from fastapi import FastAPI, Request, HTTPException
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from jose import JWTError


from app.core.logger import get_logger


logger = get_logger(__name__)



def register_exception_handlers(app: FastAPI):
    """注册全局异常处理器"""


    @app.exception_handler(HTTPException)
    async def http_exception_handler(request: Request, exc: HTTPException):
        """
        处理 HTTP 业务异常（如 400、401、404 等）


        这些是代码中主动 raise 的业务错误，不需要记录 ERROR ⽇志，
        只需返回给前端即可。
        """
        # 4xx 错误⽤ WARNING 级别（客户端问题）
        if 400 <= exc.status_code < 500:
              logger.warning(
                   "HTTP %d: %s | path=%s",
                   exc.status_code, exc.detail, request.url.path,
              )
        else:
              logger.error(
                   "HTTP %d: %s | path=%s",
                   exc.status_code, exc.detail, request.url.path,
              )


        return JSONResponse(
              status_code=exc.status_code,
              content={
                   "code": exc.status_code,
                   "message": exc.detail,
                   "data": None,
              },
        )


    @app.exception_handler(RequestValidationError)
    async def validation_exception_handler(request: Request, exc: RequestValidationError):
        """
        处理 Pydantic 参数验证异常


        当请求参数不符合 Pydantic 模型定义时触发，
        如：缺少必填字段、类型不匹配、超出范围等。
        """
        # 提取所有验证错误信息
        errors = []
        for error in exc.errors():
          field = ".".join(str(loc) for loc in error["loc"] if loc != "body")
          errors.append(f"{field}: {error['msg']}")


    logger.warning(
          "参数验证失败 | path=%s | errors=%s",
          request.url.path, errors,
    )


    return JSONResponse(
          status_code=422,
          content={
               "code": 422,
               "message": "参数验证失败",
               "data": errors,
          },
    )


@app.exception_handler(JWTError)
async def jwt_exception_handler(request: Request, exc: JWTError):
    """处理 JWT Token 解析异常"""
    logger.warning("JWT 验证失败 | path=%s | error=%s", request.url.path, str(exc))


    return JSONResponse(
          status_code=401,
          content={
               "code": 401,
               "message": "⽆效的认证凭据",
               "data": None,
          },
    )


@app.exception_handler(Exception)
async def general_exception_handler(request: Request, exc: Exception):
    """
    处理所有未预期的异常（兜底处理器）


    这是最后⼀道防线：任何没有被上⾯三个处理器捕获的异常都会到这⾥。
    必须记录完整的错误堆栈，但返回给前端时隐藏细节（安全考虑）。
    """
    # 使⽤ exc_info=True 记录完整堆栈信息
    logger.error(
          "未处理的异常 | path=%s | method=%s | error=%s",
          request.url.path, request.method, str(exc),
          exc_info=True,      # 关键：记录完整堆栈
    )


    return JSONResponse(
          status_code=500,
          content={
               "code": 500,
               "message": "服务器内部错误",
               "data": None,
              },
          )


关键知识点讲解：

 1. @app.exception_handler(异常类型) ：FastAPI 装饰器，当代码中抛出指定类型的异常时，⾃动调⽤对应的
     处理函数

 2. exc_info=True ：在 logger.error() 中使⽤此参数，会⾃动记录完整的 Python 异常堆栈
     （traceback），便于排查问题

 3. 安全原则：500 错误返回给前端时不暴露 str(exc) 详情，防⽌敏感信息泄露（如数据库连接字符串）

 4. 统⼀响应格式：所有异常都返回 {code, message, data} 格式，与正常响应保持⼀致

四、API 请求⽇志记录中间件
4.1 中间件⼯作原理
中间件（Middleware）是插⼊到请求处理流程中的⼀层"过滤器"，每个 API 请求都会经过它：


 客户端请求
      │
      ▼
 ┌──────────────────────────────────────┐
 │   请求⽇志中间件（记录请求开始时间）                │

4.2 创建中间件⽬录和模块
先创建⽬录：

⽂件： backend/app/middleware/request_logger.py


 """
 API 请求⽇志中间件


 职责：
     - 记录每个 API 请求的⽅法、路径、客户端 IP
     - 记录请求耗时（毫秒）
     - 记录响应状态码
     - ⾃动跳过健康检查等⾼频接⼝（避免⽇志噪声）


 注册⽅式（在 main.py 中）：
     from app.middleware.request_logger import RequestLogMiddleware
     app.add_middleware(RequestLogMiddleware)
 """


 import time


 from starlette.middleware.base import BaseHTTPMiddleware
 from starlette.requests import Request


 from app.core.logger import get_logger


 logger = get_logger("request")


 # 不记录⽇志的路径前缀（⾼频或⽆意义请求）
 SKIP_PATHS = [
       "/docs",
       "/redoc",
       "/openapi.json",
       "/favicon.ico",
       "/api/health",        # 健康检查频率⾼，不记录
 ]
class RequestLogMiddleware(BaseHTTPMiddleware):
    """API 请求⽇志中间件"""


    async def dispatch(self, request: Request, call_next):
        """
        拦截每个请求，记录⽇志


        Args:
              request: 当前 HTTP 请求
              call_next: 调⽤链中的下⼀个处理器
        """
        # 检查是否需要跳过
        path = request.url.path
        if any(path.startswith(skip) for skip in SKIP_PATHS):
              return await call_next(request)


        # 记录请求开始
        method = request.method
        client_ip = request.client.host if request.client else "unknown"


        # 获取请求体⼤⼩（如果有）
        content_length = request.headers.get("content-length", "0")


        logger.info(
              "→ %s %s | ip=%s | size=%s",
              method, path, client_ip, content_length,
        )


        # 记录开始时间
        start_time = time.time()


        # 调⽤下⼀个处理器（执⾏业务逻辑）
        response = await call_next(request)


        # 计算耗时
        duration_ms = (time.time() - start_time) * 1000


        # 记录请求结束
        status_code = response.status_code


        # 根据状态码选择⽇志级别
        if status_code >= 500:
              logger.error(
                  "← %s %s | status=%d | %.1fms",
                  method, path, status_code, duration_ms,
              )
        elif status_code >= 400:
              logger.warning(
                  "← %s %s | status=%d | %.1fms",
                  method, path, status_code, duration_ms,
              )
        else:
              logger.info(
                  "← %s %s | status=%d | %.1fms",
                  method, path, status_code, duration_ms,
              )


          return response


关键知识点讲解：

 1. BaseHTTPMiddleware ：Starlette 提供的中间件基类，只需重写 dispatch ⽅法即可

 2. call_next(request) ：调⽤链中的下⼀个处理器。在它之前记录"请求开始"，在它之后记录"请求结束"

 3. time.time() ：通过前后时间差计算请求耗时

 4. SKIP_PATHS ：跳过 Swagger ⽂档和健康检查请求，避免⽇志被⾼频⽆⽤信息刷屏

6.2 创建前端错误上报模块

 /**
  * 前端全局错误监控与上报
  *
  * 职责：
  *     - 捕获 JavaScript 运⾏时错误
  *     - 捕获未处理的 Promise 异常
  *     - 捕获 Vue 组件渲染错误
  *     - 将错误信息上报到后端（⽤于分析和告警）
  *
  * 使⽤⽅式（在 main.js 中）：
  *     import { setupErrorReporting } from "@/utils/errorReporter";
  *     setupErrorReporting(app);
  */


 import { ElMessage } from "element-plus";


 // ── 错误上报地址 ──────────────────────────────────────
 // Day 4 先上报到控制台 + 本地存储，Day 11 接⼊后端 API
 const REPORT_TO_BACKEND = false; // ⽣产环境设为 true
 const REPORT_API = "/api/errors/report";


 /**
  * 上报错误信息
  * 当前实现：输出到控制台 + 存⼊ localStorage
  * ⽣产环境可发送到后端⽇志收集服务
  */
 function reportError(errorInfo) {
      // 1. 控制台输出（开发调试）
     console.error("[ErrorReporter]", errorInfo);


     // 2. 存⼊本地存储（最近 50 条错误）
     try {
          const errors = JSON.parse(localStorage.getItem("error_logs") || "[]");
          errors.push({
              ...errorInfo,
              timestamp: new Date().toISOString(),
              url: window.location.href,
              userAgent: navigator.userAgent,
          });
          // 只保留最近 50 条
          if (errors.length > 50) {
              errors.splice(0, errors.length - 50);
          }
          localStorage.setItem("error_logs", JSON.stringify(errors));
     } catch (e) {
          // localStorage 写⼊失败不影响程序运⾏
          console.warn("ErrorReporter: localStorage 写⼊失败", e);
     }


     // 3. 上报到后端（⽣产环境启⽤）
     if (REPORT_TO_BACKEND) {
          fetch(REPORT_API, {
              method: "POST",
              headers: { "Content-Type": "application/json" },
              body: JSON.stringify(errorInfo),
          }).catch(() => {
              // 上报失败时静默处理，不能因为上报失败⽽再次报错
          });
     }
}


/**
    * 初始化全局错误监控
    * @param {import('vue').App} app - Vue 应⽤实例
    */
export function setupErrorReporting(app) {
     // ── 1. Vue 组件错误 ────────────────────────────────
     app.config.errorHandler = (err, instance, info) => {
          reportError({
              type: "vue_error",
              message: err.message,
              stack: err.stack,
              component: info, // 错误发⽣所在组件的⽣命周期钩⼦
          });


          // 给⽤户友好提示
          ElMessage.error("⻚⾯渲染出错，请刷新重试");
     };


     // ── 2. JavaScript 运⾏时错误 ────────────────────────
     window.onerror = (message, source, lineno, colno, error) => {
          reportError({
            type: "js_error",
            message: message,
            source: source,
            lineno: lineno,
            colno: colno,
            stack: error?.stack,
          });
     };


     // ── 3. 未捕获的 Promise 异常 ────────────────────────
     window.onunhandledrejection = (event) => {
          reportError({
            type: "promise_rejection",
            message: event.reason?.message || String(event.reason),
            stack: event.reason?.stack,
          });


          // 阻⽌默认的浏览器控制台输出（已⾃⾏处理）
          event.preventDefault();
     };


     console.log("[ErrorReporter] 全局错误监控已启⽤");
 }

⼗、前端单元测试基础（Vitest）
10.1 什么是 Vitest
Vitest 是 Vite ⽣态的测试框架，与 Jest 语法兼容，但基于 Vite 构建，速度更快。


 ┌──────────────────────────────────────────────────────────────┐
 │                 Vitest vs Jest 对⽐                                 │
 ├─────────────────┬────────────────────────────────────────────┤
 │ 构建⼯具          │ Vitest 基于 Vite（ESM 原⽣）                    │
 │                │ Jest 基于 Webpack/Babel                            │
 ├─────────────────┼────────────────────────────────────────────┤
 │ 配置             │ 复⽤ vite.config.js，零额外配置                  │
 │                │ 需要独⽴的 jest.config.js                         │
 ├─────────────────┼────────────────────────────────────────────┤
 │ 速度             │ 利⽤ Vite HMR，增量测试极快                        │
 │                │ 每次全量编译                                       │
 ├─────────────────┼────────────────────────────────────────────┤
 │ 语法             │ 与 Jest 完全兼容（describe/it/expect）          │
 └─────────────────┴────────────────────────────────────────────┘


Vitest 测试的核⼼意义：⽤⾃动化代码证明你的代码是正确的，并防⽌未来的修改破坏已有的功能。

为什么需要测试

想象⼀个场景：


 开发阶段：
     你写了⼀个登录功能 → ⼿动测试了⼏次 → 感觉没问题 → 提交代码


 ⼀周后：
     同事修改了 Token 存储逻辑 → 没意识到影响登录 → 登录功能悄悄坏了


 上线后：
     ⽤户反馈登录失败 → 排查问题花了 2 ⼩时 → ⽤户流失


如果有了测试：


 开发阶段：
     你写了登录功能 + 登录测试 → 测试通过 → 提交代码


 ⼀周后：
     同事修改 Token 存储 → 运⾏测试 → 测试失败 → ⽴即发现问题 → 修复


 上线后：
     所有测试通过 → 放⼼发布 → 零事故


测试的五⼤价值
 ┌─────────────────────────────────────────────────────────────────┐
 │                    测试的五⼤价值                                       │
 ├──────────────────────┬──────────────────────────────────────────┤
 │   1. 防⽌回归缺陷        │ 修改代码后，测试⾃动验证没有破坏旧功能                     │
 ├──────────────────────┼──────────────────────────────────────────┤
 │   2. 快速定位问题        │ 测试失败时，精确告诉你哪⾏代码出了问题                    │
 ├──────────────────────┼──────────────────────────────────────────┤
 │   3. 提升代码质量        │ 写测试时会发现代码设计缺陷，迫使你重构                    │
 ├──────────────────────┼──────────────────────────────────────────┤
 │   4. ⽂档作⽤          │ 测试⽤例本身就是活⽂档，展示代码的使⽤⽅式                   │
 ├──────────────────────┼──────────────────────────────────────────┤
 │   5. 加速开发          │ 不⽤反复⼿动测试，可以快速尝试各种⽅案                     │
 └──────────────────────┴──────────────────────────────────────────┘


覆盖率的意义


 覆盖率就像⼀张地图，告诉你：
     ✓ 哪些代码被测试覆盖了（绿⾊区域）
     ✗ 哪些代码没有被测试覆盖（红⾊区域）


 但要注意：
     ⾼覆盖率 ≠ ⾼质量测试
     覆盖率只是⼀个参考指标，不是⽬标本身


 真正重要的是：
     测试是否覆盖了关键业务场景？
     测试是否覆盖了边界条件？
     测试是否覆盖了错误路径？

原因：在错误⽬录执⾏ pytest，或虚拟环境未激活。

解决：
 cd backend            # 确保在 backend ⽬录下
 source .venv/bin/activate
 pytest



Q3：pytest 报错 sqlite3.OperationalError: no such table:
users
原因：测试数据库未正确初始化。

排查：检查 conftest.py 中是否正确导⼊了 db_models ：


 # conftest.py 中必须有这⾏导⼊
 from app.entity import db_models   # noqa: E402, F401



Q4：中间件注册后请求报错 TypeError: object is not callable
原因：中间件类没有正确继承 BaseHTTPMiddleware 。

解决：确认 RequestLogMiddleware 继承⾃ starlette.middleware.base.BaseHTTPMiddleware 。


Q5：前端 Vitest 运⾏报错 Cannot find module
原因：测试⽂件中的 @/ 路径别名未正确解析。

解决：确认 vite.config.js 中的 resolve.alias 配置正确，且 test 配置中的 include 模式正确。


Q6：健康检查详细接⼝中 PostgreSQL 报错
原因：SQLAlchemy 2.x 的 execute ⽅法签名变更。

解决：修改 health.py 中的 PostgreSQL 检查逻辑：


 # 替代⽅案：使⽤ text() 执⾏原⽣ SQL
 from sqlalchemy import text
 db = SessionLocal()
 db.execute(text("SELECT 1"))
 db.close()



Q7：前端 ElMessage 未定义报错
原因： errorReporter.js 中引⼊ ElMessage 但 Element Plus 尚未加载。

解决：确保 setupErrorReporting(app) 在 app.use(ElementPlus) 之后调⽤，或将 errorHandler 中的
ElMessage.error 改为可选调⽤：
 try {
     ElMessage?.error("⻚⾯渲染出错，请刷新重试");
 } catch {
     // Element Plus 未加载时静默处理
 }



Q8：pytest 测试间数据冲突（重复⽤户名）
原因：SQLite 测试数据库在整个测试会话期间共享，前⼀个测试创建的⽤户在后续测试中仍存在。

解决：每个测试⽤例使⽤唯⼀的⽤户名（如加随机后缀），或在 conftest.py 中使⽤ function 作⽤域的
fixture 来重置数据库：


 @pytest.fixture(autouse=True)
 def clean_database(db_session):
       """每个测试后清理数据"""
       yield
       # 回滚所有未提交的更改
       db_session.rollback()
       # 清理所有表数据
       for table in reversed(Base.metadata.sorted_tables):
           db_session.execute(table.delete())
       db_session.commit()
