# 基于YOLOv11的果蔬病害智能检测系统 - 数据库设计文档

## 一、设计总览

### 1.1 设计思路

本系统面向果蔬病害智能检测场景，涵盖用户管理、权限控制、数据集管理、病害检测、模型训练、智能体对话和系统审计等核心业务模块。数据库设计遵循以下原则：

- **多租户数据隔离**：所有业务数据通过 `user_id` 与用户关联，确保不同用户的数据相互隔离
- **RBAC权限控制**：采用标准的五表RBAC模型，支持细粒度的权限管理和角色组合
- **灵活扩展性**：对不确定结构的参数（如数据增强配置、工具调用参数）使用JSON字段存储
- **审计追踪能力**：记录所有关键操作日志，满足农业科研和生产可追溯性要求
- **性能优化**：在常用查询字段上建立索引，提升查询效率

| 模块 | 表名 | 职责 |

### 1.3 表关系清单

1. `users` ⟶ 1:N ⟶ `user_roles` — 用户可拥有多个角色
2. `roles` ⟶ 1:N ⟶ `user_roles` — 角色可被多个用户拥有
3. `roles` ⟶ 1:N ⟶ `role_permissions` — 角色可拥有多个权限
4. `permissions` ⟶ 1:N ⟶ `role_permissions` — 权限可被多个角色拥有
5. `users` ⟶ 1:N ⟶ `datasets` — 用户可创建多个数据集
6. `datasets` ⟶ 1:N ⟶ `dataset_images` — 数据集包含多张图片
7. `dataset_images` ⟶ 1:N ⟶ `dataset_labels` — 每张图片有多个标注
8. `users` ⟶ 1:N ⟶ `detection_tasks` — 用户可创建多个检测任务
9. `crop_categories` ⟶ 1:N ⟶ `detection_tasks` — 品类下有多个检测任务
10. `detection_tasks` ⟶ 1:N ⟶ `detection_results` — 检测任务包含多个检测结果
11. `model_versions` ⟶ 1:N ⟶ `detection_tasks` — 模型版本可用于多个检测任务
12. `users` ⟶ 1:N ⟶ `training_tasks` — 用户可创建多个训练任务
13. `crop_categories` ⟶ 1:N ⟶ `training_tasks` — 品类下有多个训练任务
14. `training_tasks` ⟶ 1:N ⟶ `training_metrics` — 训练任务包含多个Epoch指标
15. `training_tasks` ⟶ 1:N ⟶ `model_versions` — 训练任务产出多个模型版本
16. `crop_categories` ⟶ 1:N ⟶ `model_versions` — 品类下有多个模型版本
17. `users` ⟶ 1:N ⟶ `chat_sessions` — 用户可拥有多个对话会话
18. `chat_sessions` ⟶ 1:N ⟶ `chat_messages` — 会话包含多条消息
19. `users` ⟶ 1:N ⟶ `operation_logs` — 用户有多个操作日志

**表数量统计**：16张

**模块划分**：用户权限（5）| 果蔬品类与病害（2）| 数据集管理（3）| 检测业务（2）| 模型管理（3）| 智能体（2）| 系统运维（1）


## 二、详细表结构设计

### 模块1：用户与权限（RBAC）

#### 1. `users` — 用户表

存储系统用户的基本信息，是系统的核心实体之一。

| 字段名 | 类型 | 约束 | 说明 |
| :--- | :--- | :--- | :--- |
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 用户ID |
| `username` | String(50) | UNIQUE, NOT NULL, INDEX | 用户名 |
| `email` | String(100) | UNIQUE, NOT NULL, INDEX | 邮箱 |
| `hashed_password` | String(255) | NOT NULL | 加密密码（bcrypt哈希） |
| `phone` | String(20) | NULL | 手机号 |
| `avatar` | String(500) | NULL | 头像URL |
| `is_active` | Boolean | DEFAULT=True | 账号是否启用 |
| `is_superuser` | Boolean | DEFAULT=False | 是否超级管理员 |
| `last_login_at` | DateTime | NULL | 最后登录时间 |
| `created_at` | DateTime | DEFAULT=now() | 创建时间 |
| `updated_at` | DateTime | DEFAULT=now(), ONUPDATE=now() | 更新时间 |

**关联关系**：
- `user_roles` — 一对多，用户拥有的角色
- `datasets` — 一对多，用户创建的数据集
- `detection_tasks` — 一对多，用户执行的检测任务
- `training_tasks` — 一对多，用户发起的训练任务
- `chat_sessions` — 一对多，用户的对话会话
- `operation_logs` — 一对多，用户的操作日志


#### 2. `roles` — 角色表

定义系统中的角色，采用RBAC模型。

| 字段名 | 类型 | 约束 | 说明 |
| :--- | :--- | :--- | :--- |
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 角色ID |
| `name` | String(50) | UNIQUE, NOT NULL | 角色标识：`admin`/`agronomist`/`grower`/`viewer` |
| `display_name` | String(100) | NOT NULL | 角色显示名：管理员/农技专家/种植户/访客 |
| `description` | String(500) | NULL | 角色描述 |
| `is_system` | Boolean | DEFAULT=False | 是否系统内置角色（不可删除） |
| `created_at` | DateTime | DEFAULT=now() | 创建时间 |

**关联关系**：
- `user_roles` — 一对多，角色被用户拥有
- `role_permissions` — 一对多，角色拥有的权限


#### 3. `permissions` — 权限表

定义系统中的细粒度权限。

| 字段名 | 类型 | 约束 | 说明 |
| :--- | :--- | :--- | :--- |
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 权限ID |
| `code` | String(100) | UNIQUE, NOT NULL | 权限编码：`detection:task:create` |
| `name` | String(100) | NOT NULL | 权限名称：创建检测任务 |
| `module` | String(50) | NOT NULL | 所属模块：`auth`/`detection`/`training`/`agent`/`dataset`/`system` |
| `description` | String(500) | NULL | 权限描述 |

**关联关系**：
- `role_permissions` — 一对多，权限被角色拥有


#### 4. `user_roles` — 用户-角色关联表

实现用户与角色的多对多关联。

| 字段名 | 类型 | 约束 | 说明 |
| :--- | :--- | :--- | :--- |
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 关联ID |
| `user_id` | Integer | FOREIGN KEY(`users.id`), NOT NULL, INDEX | 用户ID |
| `role_id` | Integer | FOREIGN KEY(`roles.id`), NOT NULL, INDEX | 角色ID |
| `created_at` | DateTime | DEFAULT=now() | 创建时间 |

**关联关系**：
- `user` — 多对一，关联的用户
- `role` — 多对一，关联的角色


#### 5. `role_permissions` — 角色-权限关联表

实现角色与权限的多对多关联。

| 字段名 | 类型 | 约束 | 说明 |
| :--- | :--- | :--- | :--- |
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 关联ID |
| `role_id` | Integer | FOREIGN KEY(`roles.id`), NOT NULL, INDEX | 角色ID |
| `permission_id` | Integer | FOREIGN KEY(`permissions.id`), NOT NULL, INDEX | 权限ID |
| `created_at` | DateTime | DEFAULT=now() | 创建时间 |

**关联关系**：
- `role` — 多对一，关联的角色
- `permission` — 多对一，关联的权限

## 三、关键设计决策说明

| 设计点 | 方案 | 为什么这样做 |
| :--- | :--- | :--- |
| **RBAC权限** | `users`→`user_roles`→`roles`→`role_permissions`→`permissions` | 企业标准五表模型，支持细粒度权限控制，角色可灵活组合 |
| **果蔬与病害分离** | `crop_categories` + `disease_types` 两张字典表 | 解耦作物品类和病害类型，一种作物可有多种病害，一种病害可感染多种作物 |
| **数据集管理** | `datasets` + `dataset_images` + `dataset_labels` 三层结构 | 完整管理YOLO格式数据集，支持版本管理、数据预览和统计分析 |
| **检测结果双层结构** | `detection_tasks` + `detection_results` | 任务层存统计信息，结果层存每个目标，支持按类别筛选/统计 |
| **模型版本独立** | `model_versions` 独立表 | 支持训练产出+手动上传两种来源，支持版本切换和回滚 |
| **训练指标独立** | `training_metrics` 独立表 | 每Epoch一条记录，前端可直接查询绘制实时训练曲线 |
| **智能体对话** | `chat_sessions` + `chat_messages` | 会话级管理历史，消息级记录工具调用链路，支持审计 |
| **JSON字段** | `bbox`、`tool_calls`、`augment_config` 等使用JSON | 灵活存储结构化数据，避免过度拆表 |
| **操作日志** | `operation_logs` 独立表 | 企业级审计需求，记录谁在什么时间做了什么操作 |
| **软删除** | 使用 `status` 或 `is_active` 字段 | 保留历史数据，避免硬删除导致的数据丢失 |
