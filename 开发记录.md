#### 视频目标计数规则追加

- 固定间隔采样仅用于降低推理次数，不再把每个采样帧的检测数量直接累加
- 视频检测使用 Ultralytics ByteTrack 为连续出现的目标分配 `track_id`，同一目标后续帧只更新位置，不重复计数
- 目标首次出现时立即计数，不要求连续多帧确认，因此只在一帧出现的真实目标也会被统计
- 跟踪器无法返回 ID 时，使用相邻采样帧的类别和边界框 IoU 做降级去重
- `total_objects` 和 `class_counts` 表示视频内唯一目标数量；关键帧的 `object_count` 仍表示该帧检测到的目标数量
- 视频检测数据库只保存每个唯一目标首次出现的记录，避免任务状态接口按帧重复统计
- 摄像头检测暂不采用本次视频去重逻辑

#### 架构说明

采用**混合方案**而非纯 LangGraph 阻塞调用，原因：

- LangGraph 的 `ainvoke` 是阻塞调用，不适合 SSE 流式输出
- 混合方案：Supervisor 先路由 → 再调用对应子 Agent 的 `chat_stream`
- 完全保留现有 SSE 事件协议（thinking/tool_call/tool_result/text_chunk/done），前端无需任何修改

**阶段二架构：每个子 Agent 拥有独立工具集**

```
用户消息 → Supervisor 路由 → 选择子 Agent → 子 Agent 用自己的工具处理 → SSE 流式返回
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 Detection   QAAgent    AnalysisAgent
 Agent       (1个工具)   (3个工具)
 (4个工具)
```

| Agent          | 持有的工具                                                                                  | 数量 |
| -------------- | ------------------------------------------------------------------------------------------- | ---- |
| DetectionAgent | `detect_single_image`、`detect_batch_images`、`detect_zip_images_file`、`detect_video_file` | 4    |
| QAAgent        | `search_knowledge`                                                                          | 1    |
| AnalysisAgent  | `query_detection_stats`、`query_detection_history`、`query_user_list`                       | 3    |

**相比阶段一的优势：**

- 每个 Agent 的 Prompt 更短（不用描述所有工具），LLM 理解更准确
- 工具选择从 8 选 1 变为 4 选 1 / 3 选 1 / 1 选 1，误调用概率大幅降低
- 各 Agent 独立，互不干扰，可扩展性更强

通过 Swagger 测试 `POST /api/chat/stream`，完整事件链路：

```
session → thinking → tool_call → tool_result → text_chunk(×N) → done → [DONE]
```

- `session`：返回会话 ID
- `thinking`：Supervisor 正在分析请求
- `tool_call`：Agent 决定调用工具（如 detect_single_image）
- `tool_result`：工具执行结果（检测到的目标、类别、耗时等）
- `text_chunk`：LLM 逐 token 生成的回复内容（流式）
- `done`：对话结束，full_text 为完整回复
- `[DONE]`：SSE 流结束标记

#### 前端使用说明

前端检测完成后建议按这个流程接入：

1. 快捷检测或 Agent 检测完成后，拿到后端返回的 `task_id` 和检测结果。
2. 如果检测结果存在病害类别，调用 `GET /api/history/severity-questions` 获取问卷配置。
3. 在检测结果页直接展示问卷；用户可以立即填写，也可以跳过。
4. 用户提交问卷时调用 `POST /api/history/tasks/{task_id}/severity-assessment`，请求体里的 `class_name` 必须是检测结果原始英文类别，`answers` 的字段名使用问卷接口返回的 `key`。
5. 如果用户当时跳过问卷，历史列表或历史详情页根据 `risk_level == null` 或 `severity_assessments` 为空显示“未评估”，提供“补充评估”按钮，再走同一个问卷提交接口。
6. 用户可以在历史详情页修改治疗状态，调用 `PATCH /api/history/tasks/{task_id}/treatment-status`，不要把检测任务的 `completed` 当作已处理。
7. 如需天气风险，前端先获取用户授权定位，调用 `PATCH /api/history/tasks/{task_id}/location` 提供 `latitude`、`longitude`、可选 `location_name` 和 `location_source`。
8. 位置保存后调用 `GET /api/history/tasks/{task_id}/weather-risk`，后端会保存环境风险、天气摘要和建议；如果用户拒绝定位，前端应允许手动输入地点或跳过天气风险。
9. 报告预览调用 `GET /api/history/tasks/{task_id}/report`；一键下载调用 `GET /api/history/tasks/{task_id}/report/download` 默认下载 PDF，需要 HTML 时调用 `?format=html`。
10. 所有 History 接口都建议带上 `X-Display-Language: zh | en`，后端会返回对应语言的 `class_name_display`、`plant_name_display`、`disease_name_display`。

前端需要提供给后端的数据：

| 场景 | 需要提供 |
| ---- | -------- |
| 提交严重程度问卷 | `task_id`、检测结果原始 `class_name`、`answers`、可选 `additional_notes` |
| 保存位置 | `task_id`、`latitude`、`longitude`、可选 `location_name`、`location_source` |
| 更新治疗状态 | `task_id`、`status`、可选 `note` |
| 导出报告 | `task_id`，可选导出格式 `html/pdf`，不传默认 PDF |

前端重点状态判断：

| 状态 | 判断方式 | 建议交互 |
| ---- | -------- | -------- |
| 未评估 | `task.risk_level` 为空或 `severity_assessments` 为空 | 显示“补充评估” |
| 已评估 | 存在 `risk_level` 或 `severity_assessments` | 显示风险等级，提供“重新评估” |
| 待处理 | `treatment_status == "pending"` | 显示“开始处理”或“更新状态” |
| 高风险 | `risk_level in ["high", "critical"]` 或 `environment_risk_level in ["high", "critical"]` | 列表高亮并优先展示 |
| 天气未分析 | 没有 `environment_risk_level` 或为 `unavailable` | 引导用户补充位置或稍后重试 |
